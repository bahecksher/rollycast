## Full notes

### Decisions settled with the user
- **Backend:** `partyserver` (PartyKit's Cloudflare-native successor) on Cloudflare Workers + one Durable
  Object per room, deployed via Wrangler, with the `partysocket` auto-reconnecting client. The user asked
  "what about partykit?"; partyserver is the modern Cloudflare-native form and matches the spec's
  Workers+DO+Wrangler recommendation with far less boilerplate. Local dev/tests need no cloud account.
- **Cadence:** run straight through the milestones, keeping the project runnable after each.

### Toolchain (confirmed)
Node v24.15.0, npm 11.12.1 (workspaces; no pnpm/yarn), git 2.28 (Windows), wrangler 4.x via npx.

### Tech defaults chosen (see plan + decisions.md)
npm workspaces monorepo; TS strict; React 19 + Vite + React Three Fiber + `@react-three/rapier`; Zustand for
UI-agnostic room state (physics transforms in refs); Zod for shared runtime validation; procedural dice
geometry with canvas-texture numbers; secure server-side RNG via Workers Web Crypto with a pure,
unit-testable rejection-sampling core.

### Key architectural bet
Only the rolling client runs real physics for its active roll and streams transforms (≤12/s); other clients
interpolate and reconcile each die to the server's authoritative face via a minimal-arc quaternion. Avoids
cross-machine deterministic physics entirely.
